在一个理想的猫语翻译器里，协议层和外观层必须明确分开。协议层负责把任意 Unicode 文本安全地转成 UTF-8 字节，再经过 raw 或 zstd-dict 选择、frame 打包、base152 digit 切片和固定 token 表映射，最后得到一串可以公开传播的猫语。外观层则负责把这串猫语变得更像猫在说话，例如加上全角波浪号、感叹号和稳定的节奏停顿。这样做的好处是：无论页面看起来多可爱，底层的数据仍然能够严格、稳定、无损地来回转换。

浏览器端实现时，最重要的工程原则是不要阻塞主线程。用户在输入框里粘贴一段比较长的文本时，真正执行压缩、frame 打包和 token 映射的逻辑应该跑在 Web Worker 里，这样页面仍然能立刻响应点击、切换方向和复制结果。第一次加载 wasm 和字典时也应该在 worker 中完成初始化，并把 ready 状态回传给主线程。主线程只需要关心界面状态、错误文案和最终结果展示。

对于 GitHub Pages 这样的静态托管目标，所有必须的资产都应该在构建时就被打包好或者以相对路径发布，包括 zstd.wasm、dict_v1.bin、CSS、SVG 装饰和入口脚本。不要依赖运行时后端，不要假设固定的根路径，也不要把协议逻辑藏到服务器里。只要用户能打开页面，所有编解码行为就应该在本地浏览器里完成。
